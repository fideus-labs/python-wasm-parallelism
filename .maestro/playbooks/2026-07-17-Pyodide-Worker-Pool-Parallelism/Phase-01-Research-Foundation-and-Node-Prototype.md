# Phase 01: Research, Foundation, and Node.js Parallel Prototype

This phase researches the technical landscape (Pyodide parallelism, dask scheduler internals, multiprocessing-on-WASM constraints, and the @fideus-labs/worker-pool API), scaffolds the TypeScript project, and delivers a working end-to-end prototype: a pool of Web Workers, each running its own Pyodide interpreter, executing CPU-bound Python tasks in parallel under Node.js with a measurable speedup over serial execution. Everything runs autonomously — no user input required.

- [x] Implement the Pyodide worker script `src/worker/pyodide-worker.ts` with a one-request/one-response message protocol (the worker-pool task contract sets `onmessage` per task, so each task is exactly one message exchange):
  - On each message `{ id, kind: 'exec', code, globals?, packages? }`: lazily boot Pyodide on first use (`loadPyodide()` from the `pyodide` npm package — in Node it resolves the bundled WASM from node_modules; keep the interpreter in module scope so recycled workers skip the boot), load any `packages` not yet loaded via `pyodide.loadPackage` with a micropip fallback, run the code with `pyodide.runPythonAsync`, and post back `{ id, ok: true, result, bootMs, execMs }` with the result converted to a structured-clone-safe value (`toJs` with dict converter, or JSON round-trip)
  - On error, post `{ id, ok: false, error: { message, stack, pythonTraceback } }` — never let the worker throw unhandled
  - Support `kind: 'ping'` returning interpreter status (booted or not, loaded packages list) for tests and diagnostics
